Syntax reference

DBML input format, @layout hints, and CLI options.

Tables

Table users {
  id         int       [pk]
  email      varchar   [not null, unique]
  role       varchar   [default: 'member']
  created_at timestamp
}

Table names are case-sensitive. Column types are free-form strings — dbsketch displays them as written and does not validate them.

Column attributes

Attributes appear in square brackets after the column type, comma-separated.

AttributeDescription
[pk]Primary key.
[not null]Not-null constraint (display only).
[unique]Unique constraint (display only).
[increment]Auto-increment / serial (display only).
[default: value]Default value — quoted string or bare literal.
[ref: > table.col]Inline foreign key — see References.
[note: 'text']Column annotation (ignored by the layout).

References

Relationships can be declared inline or as standalone Ref blocks. The arrow sets the cardinality.

// Standalone declarations
Ref: posts.user_id  >  users.id       // many-to-one  (FK → PK)
Ref: users.id       <  posts.user_id  // one-to-many  (PK ← FK)
Ref: users.id       -  profiles.id    // one-to-one

// Inline, inside a Table block
Table posts {
  user_id int [ref: > users.id]
}

Schema-qualified names (public.users) are supported and rendered with the schema prefix stripped.

Layout hints

The optional @layout block fine-tunes placement without changing the schema. Each hint affects exactly one pipeline stage.

pin

Fix an entity to a specific grid column (and optionally row).

@layout {
  pin regions   at col 0
  pin stores    at col 1
  pin employees at col 2
  pin shifts    at col 3
}

center

Place a high-degree hub at the center with FK neighbors fanned to each side. Edges flow both left and right.

@layout {
  // explicit side assignment
  center sales { left: date_dim, customer_dim  right: product_dim, store_dim }

  // auto-distribute neighbors
  center sales_fact
}

preserve_order

Skip the column-reorder optimizer for specific entities, or for all entities.

@layout {
  preserve_order events, users  // named entities only
  preserve_order                // all entities
}

cluster

Split a large schema into labelled sub-diagrams. Cross-cluster foreign keys become inline annotations instead of edges.

@layout {
  cluster Auth    { users, sessions, tokens }
  cluster Billing { plans, subscriptions, invoices }
  cluster Content { posts, comments, tags }
}

SQL input

Pass a .sql file to the CLI or select SQL mode in the playground. Supported dialects: postgres (default — also handles SQLite), mysql, mssql, snowflake.

CREATE TABLE users (
  id    SERIAL PRIMARY KEY,
  email VARCHAR(255) NOT NULL UNIQUE
);

CREATE TABLE posts (
  id      SERIAL PRIMARY KEY,
  user_id INT REFERENCES users(id),
  title   TEXT NOT NULL
);

Foreign keys from REFERENCES or FOREIGN KEY … REFERENCES are extracted automatically. When none are present, dbsketch infers relationships from matching column names (user_id → users.id).

CLI

# Install
npm install -g @dbsketch/cli
pip install dbsketch            # Python wrapper, bundles Node

# Basic usage
dbsketch schema.dbml            # DBML file → stdout
dbsketch schema.sql             # SQL file (auto-detected by extension)
cat schema.sql | dbsketch --sql --dialect=mysql

# Display options
dbsketch schema.dbml --no-types     # hide column types
dbsketch schema.dbml --no-columns   # name-only boxes (relationship overview)

# SVG output
dbsketch schema.dbml --svg
dbsketch schema.dbml --svg --theme=dark

# Markdown pipeline — renders dbsketch comments in-place
dbsketch --render-markdown README.md

Library API

import { compile, compileSql, compileSvg, compileSqlSvg } from '@dbsketch/core';

// DBML → diagram string
const diagram = compile(dbml, { showTypes: true, showColumns: true });

// SQL → diagram string
const diagram = compileSql(sql, 'postgres', { showTypes: true });

// DBML → SVG string
const svg = compileSvg(dbml, { theme: 'light' });   // or 'dark'

// SQL → SVG string
const svg = compileSqlSvg(sql, 'postgres', { theme: 'dark' });
